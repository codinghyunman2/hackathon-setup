# CLAUDE.md — {Project name}

> Project rules and context for Claude Code. Auto-generated by `/hackathon-setup`. Edit freely. For product details, see `docs/PRD.md` — that is the source of truth.

## Project Overview

One short paragraph: what this project does, who runs it, and why it exists. Written so a new collaborator (human or AI) understands the purpose in 10 seconds. **Keep to 3–4 sentences max.**

## Working Style

- **Conversation language**: Korean. All explanations, plan steps, and prose to the user happen in Korean.
- **File / code / comment language**: English. Function names, file names, commit messages, inline comments all in English.
- **User is non-developer** (default for hackathon setups) — prefer no-code (n8n) or single-script solutions. Avoid microservices, Docker, framework boilerplate, multi-module project layouts.
- **Verify before assuming external access**. Before writing API integration code, ask the user to confirm they have the relevant credential. Do not invent setup steps.
- **When in doubt about an ambiguous requirement, ask one clarifying question** rather than guess silently.
- **After significant changes, summarize what changed in Korean** in 1–2 sentences.

## References

- `docs/PRD.md` — full product requirements (Section 0 has Hand-off rules for `/plan` and downstream agents — read them first)
- {External API doc / guide URL collected from user in Step 4a — keep URL verbatim}
- {Additional URL...}
- {any other files generated later by /plan}

## Translation guidance

- **Total file length: ≤ 50 lines.** If the Korean draft is longer, compress until it fits. This is a hard cap, not a target.
- The "Working Style" section bullets are mandatory and copied roughly verbatim — they are what makes Claude behave well for a non-developer user in subsequent sessions.
- "Conversation language: Korean / File language: English" is non-negotiable for this audience — always include both.
- Do not list tools or frameworks the user did not mention. Leave that to `/plan`.
- Do not duplicate PRD content (Tech Constraints, Success Criteria, Out of Scope, Primary User details). Point to the PRD instead.
- The References section must contain at least one entry: `docs/PRD.md`. External URLs from Step 4a go below that.
