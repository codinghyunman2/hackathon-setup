```markdown
# CLAUDE.md — Weekly Marketing Summary Bot

> Project rules and context for Claude Code. Auto-generated by `/hackathon-setup`. Edit freely.

## Project Overview

A Python script (or n8n workflow) that runs every Monday 9 AM KST, pulls last week's ad performance from Meta Ads, Google Ads, and Airbridge, and posts a formatted summary to the `#marketing-weekly` Slack channel. Built for an internal marketing team of 4. Non-interactive — manager reviews the message and clicks a "데이터 확인 완료" button if correct.

## Primary User

- Role: Performance Marketing Manager (non-developer, comfortable with Google Sheets and Slack)
- Headcount benefiting: 1 manager + 1 team lead (reader)
- Tech background: zero programming experience; expects to operate this without writing code after handoff

## Tech Constraints

- **Non-developer ownership** — prefer no-code (n8n self-hosted, Zapier) or a single well-commented Python file over a multi-module project. If Python is chosen, keep it under 300 lines in a single file, with environment variables documented in a sibling `README.md`.
- API credentials must live in environment variables only. Never in code, never in Slack messages, never in logs.
- Must run on the team's existing internal scheduler (cron-compatible). No new infrastructure provisioning.
- All Slack message construction must use Block Kit (not plain markdown) so the "데이터 확인 완료" button can be attached.

## Success Criteria

- Bot runs automatically every Monday 9:00 AM KST for 4 consecutive weeks with zero manual edits before the message is posted.
- Manager spends under 5 minutes on Monday morning reporting (down from 1.5 hours).
- If any of the 3 data sources fails, manager receives a DM alert within 5 minutes of the scheduled run, and the public channel message is NOT posted.

## Out of Scope

- Real-time alerts or streaming
- Historical comparisons beyond last-week-vs-week-before
- Channels other than `#marketing-weekly`
- Mobile push notifications
- Web dashboard or UI surface
- Anomaly detection or automated insights — bot reports raw numbers only

## Working Style Notes

- All conversations with the user (in any future Claude Code session) should happen in Korean.
- Code, configuration files, and inline comments stay in English.
- When suggesting libraries or services, prefer ones with Korean documentation or active Korean community support (the manager will maintain this after handoff).
- Surface every assumption explicitly — the user prefers being asked over being guessed at.

## Hand-off Notes for `/plan` and Downstream Agents

> This section is read by `/plan` and any other Claude Code workflow opening this project. Follow these rules unless the user overrides them in conversation.

- **User is non-developer.** Default to no-code or single-file Python solutions. Avoid microservices, Docker, framework boilerplate, or multi-module project layouts unless the user explicitly asks for them.
- **Explain in Korean, code in English.** Plan steps, rationale, and any prose explanation must be in Korean. File names, function names, comments, and commit messages stay in English.
- **Verify before assuming external access.** Before writing API integration code, ask the user to confirm they have the relevant credentials. Do not invent setup steps for them — link to the official onboarding doc instead.
- **Prefer recoverable changes.** When proposing a plan, sequence steps so the user can test after each one. Avoid bundles that only validate at the end.
- **Quality bar for "done".** Match the Success Criteria in this file, not a generic definition of done.

## References

- `docs/PRD.md` — full product requirements
- Slack Block Kit reference: https://api.slack.com/block-kit
- Airbridge Raw Data Export docs: https://help.airbridge.io/en/articles/raw-data-export
- Meta Marketing API: https://developers.facebook.com/docs/marketing-api
- Google Ads API: https://developers.google.com/google-ads/api

> Edit this file directly when project constraints change. Claude Code reads it at every session start.
```

## What makes this example "100점 CLAUDE.md"

1. **Project Overview is 1 paragraph, not 1 sentence** — gives Claude enough context to make independent calls without re-asking.
2. **"Non-developer ownership" is the first tech constraint** — every later code suggestion will be filtered through this lens automatically.
3. **Success Criteria are testable** — "4 consecutive weeks", "under 5 minutes", "within 5 minutes of scheduled run" — not "works well".
4. **Out of Scope is 6 bullets** — non-developers leak scope; explicit denial protects future sessions.
5. **Working Style Notes capture the multilingual policy** — conversation Korean, code English. This is the rule `/plan` and downstream agents will follow without re-deriving.
6. **References section has 5 entries** — 1 internal (PRD), 4 external API docs. Verbatim URLs survive translation and load directly when `/plan` reads CLAUDE.md.

When the user's answers are sparser, keep this structure but use fewer bullets per section. A 50-line CLAUDE.md is fine — what matters is that every section is non-empty and every bullet is specific.
