# Example: Completed `CLAUDE.md` — Weekly Marketing Summary Bot

> Read this in Step 4 BEFORE drafting the user's CLAUDE.md. Quality goal: stay under 50 lines and point to the PRD for everything else. Pair this with `example_prd.md` — Hand-off rules now live in PRD Section 0, not here.

---

```markdown
# CLAUDE.md — Weekly Marketing Summary Bot

> Project rules and context for Claude Code. Auto-generated by `/hackathon-setup`. Edit freely. For product details, see `docs/PRD.md` — that is the source of truth.

## Project Overview

An n8n workflow that runs every Monday 9 AM KST, pulls last week's Meta Ads totals, and posts a Slack summary to `#marketing-weekly`. Built for a 4-person marketing team. Manager is non-developer — operates the workflow visually in n8n after handoff.

## Working Style

- **Conversation language**: Korean. All explanations, plan steps, and prose to the user happen in Korean.
- **File / code / comment language**: English. Function names, n8n node names, commit messages, inline comments all in English.
- **User is non-developer** — prefer n8n nodes over custom code; avoid Docker, microservices, framework boilerplate.
- **Verify before assuming external access**. Before configuring API nodes, ask the user to confirm they have the credential. Do not invent setup steps.
- **When in doubt about an ambiguous requirement, ask one clarifying question** rather than guess silently.
- **After significant changes, summarize what changed in Korean** in 1–2 sentences.

## References

- `docs/PRD.md` — full product requirements (Section 0 has Hand-off rules for `/plan` — read them first)
- `docs/n8n-workflow.md` — node-by-node blueprint for the n8n graph
- Slack Block Kit reference: https://api.slack.com/block-kit
- Meta Marketing API: https://developers.facebook.com/docs/marketing-api
```

---

## What makes this example "100점 slim CLAUDE.md"

1. **Under 30 lines of actual content** — leaves room for `/plan` and other agents to inject context without blowing the session budget.
2. **Project Overview is 2 sentences** — enough to orient a new Claude Code session in 10 seconds, no more.
3. **Working Style has 6 bullets** — the rules that change *every* response Claude gives in this project. Conversation language, code language, non-developer default, credential verification, ambiguity handling, change-summarization.
4. **References has 4 entries** — internal pointers (PRD + n8n workflow doc) on top, external docs below. URLs verbatim so they survive translation.
5. **Zero duplication of PRD content** — Tech Constraints, Success Criteria, Out of Scope, Primary User details are all in PRD. CLAUDE.md doesn't repeat them.
6. **Hand-off Notes section is GONE from CLAUDE.md** — moved to PRD Section 0 where it belongs. `/plan` reads the PRD anyway, so the Hand-off rules naturally come with it.

When the user's project is sparser, keep this same shape. Do NOT add sections like "Tech Constraints" or "Out of Scope" here — those belong in the PRD. CLAUDE.md should always look this short.
